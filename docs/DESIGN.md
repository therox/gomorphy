# gomorphy — DESIGN (Phase 1)

Документ описывает архитектуру Phase 1: каркас библиотеки, формат словаря,
обратный индекс и стратегии, к которым обратится Phase 2 (реализация склонения).
В Phase 1 публичные функции возвращают `errors.New("не реализовано")`.

## 1. Источник данных

Источник — открытый словарь [OpenCorpora](https://opencorpora.org/), полный
дамп `dict.opcorpora.xml`. Лицензия — CC BY-SA. Дамп распространяется в bz2,
~30 МБ сжатого / ~500 МБ распакованного. В репозиторий он **не коммитится**:
утилита `cmd/builddict` парсит его локально в компактный `.bin`.

Из дампа отбираются леммы с тегами:

| POS-тег | Назначение |
|--------:|:-----------|
| `NOUN`  | существительные нарицательные |
| `NOUN` + `Name` | личные имена |
| `NOUN` + `Patr` | отчества |
| `NOUN` + `Surn` | фамилии |
| `ADJF`  | прилагательные, полная форма |

Глаголы, причастия, краткие прилагательные (`ADJS`), числительные и пр. —
вне области Phase 1/2.

Подтипы Name/Patr/Surn хранятся как обычные граммемы леммы. Различение
происходит по присутствию соответствующего тега в `LemmaTags`.

## 2. Формат `.bin`

Файл — конкатенация:

```
[8 байт magic "MORPHGO\x01"][gob-сериализованный internal/dict.Dict]
```

Структура `Dict` (см. `internal/dict/dict.go`):

```
Dict
├── Tags      []string                  // глобальная таблица граммем (POST/NOUN/femn/sing/...)
├── Lemmas    []Lemma                   // массив лемм
└── FormIndex map[string][]IndexEntry   // обратный индекс «нормализованная форма → леммы»

Lemma
├── ID         uint32   // равен индексу в Lemmas
├── Lemma      string   // начальная форма (lower-case)
├── LemmaTags  []Tag    // граммемы леммы (POS, род, одушевлённость, Name/Patr/Surn, Pltm, Fixd, ...)
└── Paradigm
    └── Forms []Form    // все словоформы леммы

Form
├── Text string         // словоформа (lower-case, «ё» сохранена)
└── Tags []Tag          // граммемы конкретной формы (sing/plur, nomn/gent/...)

IndexEntry
├── LemmaID  uint32     // ссылка на Dict.Lemmas
└── FormTags []Tag      // граммемы найденной словоформы
```

Дедупликация строк граммем выполняется через `Builder.InternTag`. В runtime
`Tag` — это `uint16`-индекс в `Dict.Tags`, что компактно и удобно для
будущих сравнений по равенству.

### Хранение парадигм

* **Существительные** — формы хранятся как «строка + теги» без расчёта
  основы и окончаний. Это избыточнее по объёму, но избавляет от ошибок
  морфонологии (чередования, беглые гласные). Плюралии тантум помечены тегом
  `Pltm` на лемме и имеют формы только с тегом `plur`.
* **Прилагательные (ADJF)** — в одной парадигме лежат формы для всех
  трёх родов в ед.ч. и общие формы мн.ч. Каждая форма помечена комбинацией
  `gender + number + case`. Краткие формы (ADJS) в Phase 1/2 не включаются.
* **Имена собственные** — те же поля, что и у NOUN, плюс тег
  `Name`/`Patr`/`Surn` на лемме. Несклоняемые имена/фамилии отмечены тегом
  `Fixd`; их парадигма содержит одну форму, равную лемме.

### Обратный индекс «форма → леммы»

`Dict.FormIndex` строится одновременно с парсингом. Ключ — результат
`NormalizeForm`: нижний регистр + замена `ё→е`. Это позволяет:

* находить «ёж» по запросу «еж»;
* искать без учёта регистра ввода;
* при коллизиях (одна и та же поверхностная форма для разных лемм) — хранить
  массив `[]IndexEntry`, что естественно поддерживает омонимию.

Размер индекса — `O(сумма_форм_по_всем_леммам)`. На полном OpenCorpora
ожидается порядка нескольких миллионов записей, что укладывается в
оперативную память сервера.

## 3. Согласование с числительным: правило «1 / 2–4 / 5+ / 11–14»

`Agree(word, count)` использует единственное и родительный падежи:

| count mod 100 | count mod 10 | Форма существительного |
|--------------:|-------------:|:-----------------------|
| 11–14         | —            | род. п. мн. ч. (5 яблок)            |
| —             | 1            | им. п. ед. ч. (1 яблоко)            |
| —             | 2, 3, 4      | род. п. ед. ч. (2 яблока)           |
| —             | 0, 5–9       | род. п. мн. ч. (5 яблок)            |

Сначала проверяется «11–14» — это перекрывает обычное правило для
последней цифры (например, 12 → род. мн., а не род. ед.). Отрицательные
числа берутся по модулю.

## 4. Особые группы существительных

* **Pluralia tantum** (`Pltm`): «ножницы», «сани», «сутки».
  В парадигме только формы мн.ч. Поле `Gender` для них — `GenderUnknown`.
  При вызове `Decline` со `Singular` Phase 2 будет возвращать ошибку
  «нет форм единственного числа».
* **Несклоняемые** (`Fixd`): «кофе», «метро», «Дюма».
  Парадигма содержит одну форму. `Decline` возвращает её для любых
  `Case`/`Number`. Род помечен в лемме (мужской у «кофе», средний у «метро»,
  общий `ms-f` у «Дюма»).
* **Исключения** («время», «путь», «дитя», «мать», «дочь») — обычные леммы
  с собственной парадигмой. Никакой особой обработки в коде Phase 2 не
  требуется, всё лежит в словаре.

## 5. Прилагательные: род и одушевлённость

Прилагательные ADJF склоняются в зависимости от:

* падежа,
* числа,
* рода (для ед.ч.),
* одушевлённости (только для **винительного** падежа **мужского рода**
  ед.ч. и **мн.ч.**).

Поэтому подпись:

```go
func DeclineAdj(word string, c Case, n Number, g Gender, animate bool)
```

Алгоритм Phase 2:

1. Найти лемму прилагательного (lookup в обратном индексе ИЛИ прямая лемма).
2. Выбрать форму с тегами:
   * для ед.ч.: `gender == g` + `case == c`,
   * для мн.ч.: `case == c`.
3. Особый случай: `c == Accusative` для `masc` ед.ч. либо для мн.ч.:
   * если `animate` — возвращается форма родительного падежа,
   * иначе — форма именительного падежа.

Краткие формы (ADJS) не поддерживаются.

## 6. Склонение ФИО

См. секцию **9. Phase 3: ФИО** ниже — там подробно описан реализованный
алгоритм со всеми эвристиками. Этот раздел оставлен как высокоуровневый
обзор:

* Имя (`First`), отчество (`Patronymic`) и фамилия (`Last`) склоняются
  независимо, каждое по своим правилам.
* Любое поле может быть пустым; пустые поля остаются пустыми в результате.
* Род определяется по приоритету: отчество → имя → фамилия (см. §9).
* Для отчеств и фамилий, отсутствующих в словаре, работают эвристики
  по окончанию начальной формы.

## 7. Морфологический разбор

`Parse(word string) ([]Analysis, error)`:

1. Нормализовать `word` через `NormalizeForm` (lower-case + `ё→е`).
2. Получить `dict.FormIndex[ключ]` — список `IndexEntry`.
3. Для каждой записи:
   * прочитать `Lemma` по `IndexEntry.LemmaID`;
   * собрать `Analysis`, копируя POS / падеж / число / род / одушевлённость
     из объединения `LemmaTags + FormTags`.
4. Если ключ отсутствует — вернуть **пустой срез** `[]Analysis{}` и
   `nil`-ошибку. Не `nil`-срез: вызывающему коду удобнее единообразно
   итерировать `range` по результату без проверки на `nil`.
   Угадывание по окончанию **не выполняется** в Phase 1/2.

### Омонимия

Одна форма может соответствовать нескольким леммам или нескольким разборам
в пределах одной леммы (например, «стали» — род./дат./предл. ед.ч.
существительного «сталь» **и** прош.вр. глагола «стать», который у нас
не индексирован). `Parse` возвращает все варианты, что позволяет вызывающему
коду сделать собственную дезамбигуацию по контексту.

### Регистр и «ё»

Нормализация в индексе и в lookup-е выполняется одинаково. Поэтому
`Parse("Ёж")`, `Parse("ёж")`, `Parse("еж")` дадут идентичный результат.
Поле `Lemma` в результате — в нижнем регистре, как хранится в словаре;
восстановление исходного регистра — задача потребителя (например,
капитализация по позиции в предложении).

### Слова вне словаря

Возвращается пустой срез `[]Analysis{}` и `nil`-ошибка. Эвристическое
угадывание (по окончанию, по соседним словам, по статистическим моделям) —
**вне области** этого проекта; библиотека словарная и должна давать только
подтверждённые разборы.

## 8. Phase 2: реализация

В Phase 2 публичные функции получили реальные тела. Алгоритмы единообразны:
сначала `getDict()` отдаёт глобальный словарь (см. `init.go`),
затем `dict.Lookup(word)` нормализует входную форму и возвращает все
интерпретации из обратного индекса. Дальше — точечная логика по типу
функции.

### `Init` и автозагрузка из `GOMORPHY_DICT`

Глобальный словарь хранится как один указатель под мьютексом. `Init(path)`
загружает `.bin` через `internal/dict.LoadFile` и помечает синглтон
заполненным; повторный вызов возвращает ошибку «уже инициализирован».
Если `Init` не вызвался, при первом обращении к API проверяется переменная
окружения `GOMORPHY_DICT`; пустое значение → ошибка с понятным текстом.
Никаких асинхронных перезагрузок, LRU и подобной сложности — только
указатель и мьютекс.

### `Decline(word, c, n)` и общий `declineNoun`

1. `Lookup(word)` → набор `IndexEntry`.
2. Среди интерпретаций берётся первая лемма с тегом `NOUN`.
3. Вызывается `declineNoun(d, lemma, c, n)`:
   * лемма с тегом `Fixd` (несклоняемая, например «кофе») — возвращается
     `lemma.Lemma` для любого падежа/числа;
   * лемма с тегом `Pltm` при `n == Singular` — ошибка
     «pluralia tantum, нет форм единственного числа»;
   * иначе — линейный поиск формы с тегами `caseTag(c)+numberTag(n)`
     в `lemma.Paradigm.Forms`.

`Agree` использует тот же `declineNoun`; правило выбора падежа/числа
вычисляется отдельной чистой функцией `agreeCaseNumber(count)` без
обращения к словарю — её удобно покрыть табличным тестом без I/O.

### `DeclineAdj(word, c, n, g, animate)` — винительный по одушевлённости

Алгоритм отличается лишь обработкой винительного:

* для `Singular` + `Masculine` + `Accusative`: `animate=true` → берём
  форму родительного падежа, иначе — именительного;
* для `Plural` + `Accusative` (любой `g`): аналогично, но без фильтра по роду;
* для остальных комбинаций берётся форма строго по `case+number+gender`.

Это полностью соответствует поведению русского винительного: `красного` для
одушевлённого и `красный` для неодушевлённого, `красных` для одуш. мн.ч.
и `красные` для неодуш. мн.ч.

### Правило согласования «1 / 2–4 / 5+ / 11–14»

`agreeCaseNumber(count)` берёт `|count|`, проверяет сначала исключение
11–14 (всегда `Genitive+Plural`), затем последнюю цифру:

| `n%10` | падеж/число        | пример       |
|-------:|:-------------------|:-------------|
| 1      | им. п. ед. ч.      | 1 яблоко     |
| 2,3,4  | род. п. ед. ч.     | 3 яблока     |
| 0,5–9  | род. п. мн. ч.     | 5 яблок      |

Проверка 11–14 идёт первой и перекрывает обычное правило: 12 → род. мн.,
а не род. ед.

### Нормализация падежных тегов при парсинге

В дампе OpenCorpora для слов с распределением на «канонический» и
«партитивный/местный» падеж используется пара тегов:

* `gen1` — основной родительный (есть пара `gen2` — партитив: «чашка чаю»);
* `loc1` — основной предложный (`loc2` — местный: «в саду», «на берегу»);
* `acc1` — основной винительный (`acc2` — встречается реже).

При построении `.bin` `internal/dict/parser.normalizeCaseAliases` приводит
`gen1`→`gent`, `loc1`→`loct`, `acc1`→`accs`. Это упрощает Decline/Parse
и убирает лишний словарь тегов. Партитив/местный (`gen2`/`loc2`) остаются
как есть — по основным шести падежам они не возвращаются, но видны в
исходных тегах формы при отладке.

### `Parse` — расшифровка тегов

Для каждого `IndexEntry` из `Lookup`:

1. Создаётся `Analysis` с `Lemma` и `POS` (по тегам леммы).
2. Падеж и число извлекаются из тегов формы (`parseCase`/`parseNumber`).
3. Род для NOUN копируется с тегов леммы; для ADJF — берётся с тегов формы
   (для мн.ч., где род в парадигме не различается, остаётся `GenderUnknown`).
4. `Animate` — true, если тег `anim` встретился в форме или на лемме.

`Parse` возвращает все варианты в порядке индекса; вызывающий код сам
дезамбигуирует по контексту. Слово вне словаря → пустой непустой срез
(`[]Analysis{}`) и `nil`-ошибка.

## 9. Phase 3: ФИО

`DeclineFullName(name FullName, c Case)` склоняет фамилию, имя и отчество
независимо. Любое поле `FullName` может быть пустым — пустые поля остаются
пустыми в результате (без ошибки и без обращения к словарю). Полностью
пустой `FullName{}` — единственный вход, на который функция возвращает
ошибку «пустое имя».

### 9.1. Определение рода

`detectGender` идёт по приоритету источников; каждый шаг пропускается,
если соответствующее поле пустое:

1. **Отчество** — суффиксная классификация (без обращения к словарю):

   | Суффикс | Род |
   |--------|-----|
   | `-ович`, `-евич`, `-ич`, `-ьич` | Masculine |
   | `-овна`, `-евна`, `-ична`, `-инична` | Feminine |

2. **Имя** — `Lookup(First)` среди лемм с тегом `Name`, берётся род леммы.
   Common (`ms-f`) считается «недостаточно информативным» и пропускается
   в пользу следующего источника.

3. **Фамилия** — сперва эвристика по окончанию (см. ниже), затем `Lookup`
   среди лемм с тегом `Surn`, чья начальная форма совпадает со входом.
   Эвристика проверяется первой намеренно: иначе для «Иванова» словарный
   поиск нашёл бы лемму «иванов» (масc) с формой род.п. «иванова»
   и неверно определил бы род.

   Информативные окончания эвристики:

   | Суффикс | Род |
   |--------|-----|
   | `-ов`, `-ев`, `-ёв`, `-ин`, `-ын`, `-ский`, `-цкий` | Masculine |
   | `-ова`, `-ева`, `-ёва`, `-ина`, `-ына`, `-ская`, `-цкая` | Feminine |

   Прочие окончания (Гайдай, Скрипка, Долгих, Шевченко) считаются
   неинформативными — род должен прийти от отчества или имени.

4. Все три поля пустые → ошибка `"DeclineFullName: пустое имя"`.

5. Поля заполнены, но ни один источник не дал рода → ошибка
   `"не удалось определить род"`.

### 9.2. Склонение отчеств — эвристика

Если отчества нет в словаре или у леммы нет нужной формы, используется
суффиксная парадигма. Отчества склоняются строго регулярно.

* **Мужской тип** (`-ович`/`-евич`/`-ич`/`-ьич`): отрезаем последние две
  руны «ич», добавляем падежный суффикс:
  `-ич` (Им) / `-ича` (Род) / `-ичу` (Дат) / `-ича` (Вин) / `-ичем` (Тв) /
  `-иче` (Пред).
* **Женский тип** (`-овна`/`-евна`/`-ична`/`-инична`): отрезаем последнюю
  руну «а», добавляем падежный суффикс:
  `-а` (Им) / `-ы` (Род) / `-е` (Дат) / `-у` (Вин) / `-ой` (Тв) / `-е` (Пред).

### 9.3. Склонение фамилий — эвристика

Реализована как один плоский switch по суффиксам в `declineSurnameHeuristic`.
Возвращает `(результат, true)` либо `("", false)`, если фамилия не подошла
ни под один шаблон или род не совпал с шаблоном.

| Окончание | Пол | Тип склонения |
|----------|-----|---------------|
| `-ов`, `-ев`, `-ёв`, `-ин`, `-ын` | M | Притяжательное муж.: `-/а/у/а/ым/е`. Винительный одуш. = род. |
| `-ова`, `-ева`, `-ёва`, `-ина`, `-ына` | F | Притяжательное жен.: `а/ой/ой/у/ой/ой` |
| `-ский`, `-цкий` | M | Адъективное муж. (как «русский»). После к/г/х/ц в Тв. — `им` (Достоевским). |
| `-ская`, `-цкая` | F | Адъективное жен. (как «русская»): `ая/ой/ой/ую/ой/ой` |
| `-ой`, `-ый`, `-ий` (вне `-ский`/`-цкий`) | M | Адъективное муж.: `ого/ому/им|ым/ом` (выбор `им`/`ым` — по букве перед окончанием) |
| `-ая`, `-яя` | F | Адъективное жен.: `ой|ей/.../ую|юю/...` |
| `-ых`, `-их` (Долгих, Седых) | M/F | **Несклоняемая** — возвращается как есть |
| Согласный (вне списка) | M | 2-е скл.: согласный `+ а/у/а/ом|ем/е`; на `-й`/`-ь` — `я/ю/я/ем/е` (Гайдай → Гайдая, Гайдаю...). После шипящих/ц в Тв. — `ем`. |
| `-а` (вне списка) | M/F | 1-е скл.: `а/ы|и/е/у/ой/е`; после к/г/х/ж/ш/щ/ч в Род. — `и` (Скрипка → Скрипки) |
| `-я` (вне списка) | M/F | 1-е скл.: `я/и/е/ю/ей/е`; подтип `-ия` — `я/и/и/ю/ей/и` (как «Мария») |
| `-о`, `-е`, `-у`, `-ю`, `-и`, `-ы`, `-э`, `-ё` | M/F | **Несклоняемая** — возвращается как есть |

#### Упрощение «ударные гласные»

Реальное правило русской морфологии — несклоняемые иностранные фамилии
с ударным `-а`/`-я` (Дюма, Золя). Определять ударение программно сложно
(у `gomorphy` нет акцентологического словаря), поэтому принято
**сознательное упрощение**: окончание `-а`/`-я` всегда приводит к 1-му
склонению эвристикой. Это означает, что фамилии Дюма-класса вне словаря
будут склоняться (Дюма → Дюмы, Дюме, Дюму, Дюмой, Дюме) — поведение
зафиксировано в `TestSurnameHeuristic1stDecl` и
`TestDeclineFullNameAlphaSurnameHeuristic`.

Чтобы получить корректное (несклоняемое) поведение для конкретной
иностранной фамилии на ударный `-а`, добавьте её в словарь с тегом
`Fixd` — словарный путь имеет приоритет над эвристикой. В мини-словаре
`testdata/sample.xml` Дюма помечена как `Surn+Fixd+ms-f` именно для этой
цели; тест `TestDeclineFullNameIndeclSurnameDict` подтверждает, что
такая лемма возвращается без изменений во всех падежах.

Прочие конечные гласные (`-о`, `-е`, `-у`, `-ю`, `-и`, `-ы`, `-э`, `-ё`)
эвристика честно считает несклоняемыми — это покрывает большинство
типичных случаев (Шевченко, Кобо, Дали).

### 9.4. Склонение имён

Имена ищутся в словаре среди лемм с тегом `Name`, фильтруются по
совместимому роду и склоняются через общий `declineNoun`. Имени, которого
в словаре нет — или у которого в парадигме нет нужной формы — возвращается
без изменений (несклоняемое иностранное). Эвристик для имён не строится:
имена нерегулярны (Лев → Льва, Пётр → Петра), а ошибочное склонение хуже
неизменения.

### 9.5. Регистр кириллицы

`applyCase(orig, formed)` копирует регистр первой буквы исходной формы
на результат: если первая буква `orig` — заглавная, заглавится и первая
буква `formed`. Используется `unicode.IsUpper`/`unicode.ToUpper`,
которые корректно работают с кириллицей в Go без ручных таблиц.
Остальные буквы результата остаются в нижнем регистре (как хранится
в словаре и строится эвристикой) — для типичного «Иванов»/«Анна» этого
достаточно. Для редких случаев типа «фон Бенкендорфа» выбор регистра
для каждой буквы остаётся за вызывающим кодом.

### 9.6. Сочетаемость с Common (ms-f)

Лемма с тегом `ms-f` (Дюма, сирота) совместима с любым `want` родом
в `genderCompatible`: благодаря этому `FullName{Last: "Дюма"}` без имени
успешно склоняется (точнее — возвращается как есть, поскольку Дюма помечена
`Fixd`), а `FullName{Last: "Дюма", First: "Александр"}` корректно
склоняет имя «Александр» по мужскому роду, унаследованному от имени.

## 10. Phase 4: разбор ФИО (любой падеж → им.п.)

Phase 4 добавляет два публичных API, симметричных Phase 3:

* `ToNominative(name FullName) (FullName, error)` — каждый компонент
  `name` (любой падеж) приводится к им.п. независимо.
* `ParseFullName(s string) (FullName, error)` — строка разбивается
  на компоненты Last/First/Patronymic и приводится к им.п.

В паре с `DeclineFullName` это решает типовой кейс реальных БД:
«ФИО в любом падеже → ФИО в любом другом падеже».

### 10.1. Обратная эвристика отчеств

`inversePatronymic(s) → (Nom, gender, ok)` распознаёт пару (стем, падежное
окончание):

| Тип | Стем оканчивается на | Падежные окончания | Nom-форма |
|-----|---------------------|---------------------|-----------|
| Мужской | `ич` | `«»`, `а`, `у`, `е`, `ем` | стем + `ич` |
| Женский | `иничн` / `овн` / `евн` / `ичн` | `а`, `ы`, `е`, `у`, `ой` | стем + `а` |

Стем `иничн` проверяется **раньше** `ичн` (более длинный/специфичный
шаблон): иначе «Ильиничны» сократилось бы до «Ильинича» вместо
«Ильинична».

Возвращаемый род становится подсказкой (`gHint`) для разбора фамилии.

### 10.2. Обратная эвристика фамилий

`inverseSurnameHeuristic(last, gHint) → (Nom, gender, ok)` устроен как
плоский switch по суффиксам входной формы. Порядок важен — более
специфичные шаблоны проверяются раньше:

| Суффикс входа | Распознаётся как | Nom |
|---------------|------------------|-----|
| `-ых` / `-их` | indeclinable | как есть |
| `-ская` / `-цкая` / `-ской` / `-цкой` / `-скую` / `-цкую` | F adj | стем + `ская` / `цкая` |
| `-ский` / `-цкий` / `-ского` / `-цкого` / `-скому` / `-цкому` / `-ским` / `-цким` / `-ском` / `-цком` | M adj | стем + `ский` / `цкий` |
| `-овой` / `-евой` / `-ёвой` / `-иной` / `-ыной` | F poss (однозначно) | стем + `а` |
| `-ова` / `-ева` / `-ёва` / `-ина` / `-ына` | F Nom **или** M Gen — по `gHint` | F: как есть; M: стем без `а` |
| `-ову` / `-еву` / `-ёву` / `-ину` / `-ыну` | F Acc **или** M Dat — по `gHint` | F: стем + `а`; M: стем без `у` |
| `-овым` / `-евым` / `-ёвым` / `-иным` / `-ыным` | M poss Inst | стем без `ым` |
| `-ове` / `-еве` / `-ёве` / `-ине` / `-ыне` | M poss Prep | стем без `е` |
| `-ов` / `-ев` / `-ёв` / `-ин` / `-ын` | M poss Nom | как есть |
| `-ую` / `-юю` | F adj Acc | стем + `ая` / `яя` |
| `-ая` / `-яя` | F adj Nom | как есть |
| `-ой` (вне -ск-/-цк-) | M Nom **или** F gen/dat/inst/prep — по `gHint` | M: как есть; F: стем + `ая` |
| `-ого` / `-ому` / `-ым` / `-им` / `-ом` | M adj | стем + `ой` (после прочих согласных) или `ий` (после к/г/х/ц) |
| `-ый` / `-ий` (вне -ск-/-цк-) | M adj Nom | как есть |
| `-ы` / `-и` / `-е` / `-у` (после согласной, не подпало выше) | 1-е скл. | замена последней руны на `а` |
| `-ю` (после согласной, не подпало выше) | 1-е скл. (-я) | замена `ю` на `я` |
| Прочие гласные | indeclinable | как есть |
| Согласная (не подпало выше) | M 2-е скл. Nom | как есть |

#### Двусмысленные формы и `gHint`

Окончания `-ова`, `-ову`, `-ой` — омонимичны между разными родами и
падежами. `gHint` (приходит из отчества или имени) разрешает омонимию:

* `gHint = Masculine`: `-ова` → M Gen («Иванов»), `-ову` → M Dat,
  `-ой` → M Nom («Толстой»).
* `gHint = Feminine`: `-ова` → F Nom («Иванова»), `-ову` → F Acc,
  `-ой` → F Gen/Dat/Inst/Prep («Толстая»).
* `gHint = GenderUnknown`: `-ова`/`-ову` → F Nom (более частая трактовка
  для «незнакомой» фамилии); `-ой` → M Nom (Толстой).

#### Выбор `-ой` vs `-ий` для адъективных

Формы `-ого`/`-ому`/`-ым`/`-им`/`-ом` (вне `-ск-/-цк-`) теряют
информацию, какое было Nom-окончание. Эвристика `adjMascNomEnding`:
после к/г/х/ц → `-ий` (Достоевский, Горький); иначе → `-ой` (Толстой).
Ветка `-ый` намеренно опущена — `-ый` в фамилиях редок, и `-ой` даёт
правильный результат для большинства реальных вариантов.

### 10.3. `ToNominative` — алгоритм

1. **Отчество**: `inversePatronymic` → Nom + `gHint`.
2. **Имя**: словарь (`Lookup` среди `Name`-лемм) → `Lemma` это им.п.;
   при отсутствии — оставляем как есть. Род леммы становится `gHint`,
   если ещё не задан.
3. **Фамилия**: словарь с фильтром по `gHint`, иначе
   `inverseSurnameHeuristic` с тем же `gHint`. При отсутствии `gHint`
   приоритет отдаётся лемме, чья начальная форма равна входу — иначе
   для «Иванова» был бы возвращён родительный падеж от мужской «Иванов».

Регистр первой буквы каждого компонента переносится с исходной формы
через `applyCase` (как в `DeclineFullName`).

### 10.4. `ParseFullName` — алгоритм

1. Токенизация по пробелам (`strings.Fields` после `TrimSpace`).
2. Допускается 1, 2 или 3 токена; иначе — ошибка.
3. Распознавание ролей по позиции отчества:
   * 3 токена, `Patr` в позиции [2] → русский «Last First Patr»;
   * 3 токена, `Patr` в позиции [1] → западный «First Patr Last»;
   * 3 токена, `Patr` в позиции [0] → ошибка
     («отчество не может стоять первым»);
   * 3 токена без отчества → fallback «Last First X» (неоднозначный
     разбор, пользователь увидит результат как есть);
   * 2 токена с отчеством → имя + отчество без фамилии;
   * 2 токена без отчества → если один из токенов — `Name` в словаре,
     он становится First, другой — Last; иначе по умолчанию
     «Last First» (самый частый порядок в БД);
   * 1 токен → пробуем как Patronymic, затем как Surname (словарь или
     эвристика), иначе — First.
4. Над полученным `FullName` вызывается `ToNominative`.

### 10.5. Round-trip

Связка `ParseFullName + DeclineFullName` симметрична `DeclineFullName +
ToNominative`: для любого `nom` и любого падежа `c`

```
ToNominative(DeclineFullName(nom, c)) == nom
```

— это закреплено в `TestToNominativeRoundTrip` для всех типов фамилий
из таблицы Phase 3.
